# 思宜案例：Codex 教學準備筆記

用途：保留可放入後續課程講綱的兩個核心段落。

## 1. 上述需求如果用 Codex 要怎麼做

| 需求場景 | Codex 做法 | 需要先準備 | 需要人工確認 |
| --- | --- | --- | --- |
| 每日讀 InfoCenter 狀態與新增個案 | 每日固定時間讀取「馴錢師財商顧問股份有限公司」單位內案件，和前一天快照比對，整理新增個案、狀態變更、待追蹤清單 | InfoCenter API、認證方式、單位 ID、案件列表與詳情欄位、狀態定義 | 確認清單是否符合實際工作情境，不直接對外通知 |
| 特殊欄位判斷 | 依照思宜提供的規則，標示警示帳戶、特殊境遇、欄位衝突、缺漏資料等狀況，並附上判斷原因 | 特殊欄位清單、觸發條件、對應動作、例外規則、優先順序 | 個案專業判斷與下一步聯繫仍由人決定 |
| 說明會前名單 | 說明會前讀取報名、同意書、LINE 驗證、說明會參加狀態，整理未參加或尚未確認的人 | 說明會場次、日期區間、未參加定義、輸出表格欄位 | 確認名單後再由思宜聯繫 |
| 引導員電訪追蹤 | 派案後定期讀 InfoCenter 電訪欄位與 Google Sheet 引導員通知表，判斷已完成、逾期、缺資料、兩邊狀態不一致 | 派案日、完成期限、電訪完成判斷欄位、Google Sheet 欄位、可自動更新欄位 | 自動更新前要先確認哪些欄位能寫回，並保留更新紀錄 |
| 信扶匯入檔案產出 | 依照信扶模板與欄位 mapping，從 InfoCenter 抓資料後產生 Excel/CSV，另附缺值與需人工確認清單 | 信扶匯入模板、欄位 mapping、格式規則、必填欄位、固定範圍空間 | 匯入前抽樣核對，並先用測試資料驗證 |

教學重點：Codex 先從 read-only 的查詢、比對、整理開始；涉及對外通知、正式系統寫入、個案判斷時，要設人工確認點。

## 2. 可能會用到的 Codex 功能清單

| Codex 功能 | 在思宜案例中的用法 | 課程講法 |
| --- | --- | --- |
| API 讀取 | 讀 InfoCenter 案件、個案狀態、同意書、說明會、派案與電訪欄位 | 讓 Codex 讀系統資料，但先從 read-only 開始 |
| 資料快照比對 | 比對今天與昨天有哪些新增個案或狀態變更 | 把「每天人工一直看」改成「有變化才提醒」 |
| AI 初步分類 | 判斷警示帳戶、特殊境遇、欄位衝突、缺漏資料、逾期狀態 | AI 先幫忙分流，人負責最後判斷 |
| Google Sheet 讀寫 | 讀取與更新引導員通知表、狀態欄、備註欄 | 可以自動更新，但要先講清楚哪些欄位能改 |
| Excel/CSV 產生 | 依照信扶模板產出匯入檔案 | 把複製貼上與刪欄位變成模板化產出 |
| 排程 automation | 每天、說明會前、派案後固定檢查 | Codex 可以固定時間幫忙檢查，不需要每天想起來 |
| 報告發布 | 產出 Markdown 或 HTML，讓需求、流程、規則跨裝置查看 | 適合需求討論與教學展示 |
| 權限安全檢查 | 檢查 API token、個資、公開範圍、外部寫入權限 | 先讀資料，再決定能不能自動寫入 |
| 人工審核點 | 對外通知、個案判斷、正式系統異動前確認 | Codex 負責準備，人負責決策 |

## 3. 歷程流程表

| 歷程階段 | 這一步在做什麼 | Codex 可以怎麼協助 | 產出物 |
| --- | --- | --- | --- |
| 1. 蒐集工作紀錄與逐字稿 | 先把思宜實際描述的工作內容、痛點、判斷方式整理出來 | 協助摘要逐字稿、抓出角色分工、系統來源、重複動作與痛點 | 需求摘要、角色分工、現況流程 |
| 2. 拆解現在的工作流程 | 把同意書、LINE、說明會、派案、引導員電訪、信扶匯入拆成連續步驟 | 將口語描述轉成流程圖、流程表、階段任務 | 現況流程表、痛點說明 |
| 3. 判斷哪些工作適合 Codex | 區分哪些是查資料、比對、整理、產檔，哪些是專業判斷或對外聯繫 | 依第一性原則標示可自動化、半自動、人工保留的部分 | Codex 可協助項目清單 |
| 4. 定義資料來源與規則 | 釐清 InfoCenter、Google Sheet、信扶模板、說明會資料需要哪些欄位 | 協助整理欄位 mapping、觸發規則、例外情境、人工確認點 | 資料準備清單、規則表 |
| 5. 設計 Codex 任務 | 把每日讀取、特殊欄位判斷、說明會名單、引導員追蹤、信扶匯入拆成任務 | 規劃 read-only 報告、排程 automation、Google Sheet 更新、Excel/CSV 產出 | 自動化任務規劃 |
| 6. 建立教學素材 | 把上述需求轉成同仁聽得懂的教學段落 | 產出「怎麼用 Codex 做」與「會用到哪些功能」兩個段落 | Codex 教學準備筆記 |
| 7. 發布與版本管理 | 把報告放到 GitHub 與 Vercel，讓不同裝置可以查看 | 協助 commit、push、production deploy，並保留 noindex 控制 | 公開頁面、Markdown 檔、版本紀錄 |
| 8. 回到課程整併 | 後續把每位同仁的流程用同一張表整合 | 用相同欄位統一整理不同人的需求，形成課程講綱 | 統整版課程流程表 |
