思宜需求分析與 Codex 自動化任務規劃

把訪談逐字稿轉成可討論、可驗證、可逐步自動化的工作流草案。重點不是直接替人判斷個案，而是讓每天該追蹤、該檢查、該匯入的資料浮上來。

建議先做的 MVP

  1. 每日提醒清單
  2. 欄位檢查與衝突報告
  3. 信扶上傳模板產生器

一頁摘要

思宜的工作橫跨多個系統與多個角色。最有價值的自動化，不是讓 AI 主動對外處理個案，而是把分散資訊整理成可行動清單。

核心需求

今天該看誰、該追誰、誰有新狀態

同意書、LINE 驗證、說明會、引導員電訪與匯入狀態散在不同地方，Codex 可以先扮演每日盤點與比對助手。

自動化邊界

讀取與提醒優先，外部寫入保留人工核准

LINE 發送、個案取消、資格判定、正式上傳與系統回寫都應該先停在人工確認點。

工作流程拆解

從自行報名到信扶系統匯入，可以切成四段：同意書與 LINE、初步資格與說明會、引導員電訪追蹤、信扶匯入與回寫。

1. 自行報名
網站報名後，雨虹匯入 InfoCenter 並開同意書事件，系統發送簡訊。
可產出：同意書待追清單
2. LINE 驗證
個案回覆同意書後加入官方 LINE，系統驗證姓名與電話，InfoCenter 顯示 LINE 狀態。
可產出：未驗證/新驗證清單
3. 初步判斷
思宜確認子女學齡、同住、特殊境遇、警示帳戶、是否參加說明會或看短片。
可產出：關鍵欄位提醒
4. 引導員電訪
有意願名單派案後，引導員應在期限內完成第一次聯繫與電訪，並填寫結果。
可產出：新完成/逾期未完成清單
5. 匯入信扶
思宜人工確認可匯入名單，整理欄位、上傳信扶系統，再回寫匯入日期與狀態。
可產出：上傳模板與驗證報告

需求切片

A

同意書與 LINE

目前沒有新加入或新回覆通知，需要每日列出已回覆未驗證、已驗證待回覆、未參加說明會但可能仍有意願者。

B

欄位正確性

比起自動回覆，思宜更明確需要欄位檢查：身分證與性別、警示帳戶、特殊境遇、子女學齡與同住、評論與背景欄位衝突。

C

電訪追蹤

不能只看電訪欄位有沒有值，因為思宜或其他內部角色也會填。需要用歷史紀錄、最後填寫者、派案狀態與日期一起判斷。

D

信扶匯入

目前最大手工成本在欄位刪減、貼上、對齊與匯入後回寫。這適合做成模板產生器與驗證報告。

E

提醒而非查詢

思宜要的不是多功能查詢，而是關鍵條件出現時自動浮出名單，降低她每天逐頁檢查的成本。

F

人工判斷保留

資格說明、個案意願、特殊情境處理仍需要人判斷。Codex 應該協助整理與預檢，不應直接做結論。

Codex 自動化任務規劃

建議用階段式導入，從不寫入任何外部系統的讀取型報告開始，規則穩定後再進入半自動回寫。

Phase 0：盤點與規則定義

整理 InfoCenter 匯出欄位、信扶上傳模板、引導員表、LINE 狀態與歷史紀錄樣本，建立欄位 mapping 與排除規則。

Phase 1：每日讀取型提醒報告

每天產生同意書/LINE、警示帳戶、特殊境遇、引導員新完成、逾期未完成、欄位衝突等清單。

Phase 2：資料品質檢查器

檢查身分證/性別一致性、必問欄位、評論未回填、事件未開、無意願或三通未接是否應排除。

Phase 3：信扶上傳模板產生器

依據待匯入名單與欄位對照表產生可上傳 Excel/CSV，同時輸出缺值、格式錯、需人工判斷欄位。

Phase 4：人工核准式外部系統更新

在規則穩定後，Codex 可協助開系統、預填匯入日期或狀態，但送出前需要人工確認。

提醒規則草案

清單 判斷條件 用途
今日新同意書回覆 新增同意書回覆，但尚未完成下一步 追 LINE 或說明會狀態
已回覆但未 LINE 驗證 同意書完成，LINE 狀態未綠燈 補追加入 LINE 或姓名電話驗證
警示帳戶 欄位顯示警示帳戶 優先電話聯繫
引導員新完成電訪 電訪欄位由引導員更新，且昨日沒有 提醒思宜檢查資料
逾期未完成電訪 派案超過規定天數仍無引導員電訪結果 催辦或追蹤引導員
欄位/評論衝突 背景欄位與電訪評論不一致，或評論有資料但欄位空白 人工修正與補資料

可能解決方案

先做

Google Sheet / Excel 版提醒儀表板

每天把匯出檔與引導員表放入固定資料夾，Codex 比對昨日快照並產出報告。優點是權限風險低、容易核對規則。

MVP 後

InfoCenter + Sheets 半自動監控

由 Codex 讀取 InfoCenter 與表單，保存每日快照，辨識新完成與逾期未完成。需要先確認登入與欄位穩定性。

高價值

信扶匯入模板產生器

建立 InfoCenter 到信扶模板的欄位 mapping，直接產生可上傳檔與錯誤報告，減少貼錯與欄位對齊成本。

後期

人工核准式瀏覽器助手

Codex 預填外部系統欄位，在送出前停下讓思宜確認。適合規則穩定後導入。

建議優先順序

優先 項目 原因
P0 每日引導員電訪變更/逾期提醒 思宜明確說目前沒有時間追，且引導員不主動回報。
P0 信扶上傳模板產生器 手工欄位對齊高耗時且高錯誤率。
P1 欄位正確性與衝突檢查 思宜明確覺得這塊比 LINE 自動化更適合。
P1 同意書/LINE/說明會狀態清單 有幫助，但前段仍需要大量人工判斷。
P2 自動開事件/回寫狀態 有價值，但要等規則穩定與權限確認。

需要再確認的問題

  1. InfoCenter 中各狀態與欄位的實際名稱是什麼？
  2. 歷史紀錄是否能看到更新者、更新時間、更新欄位、更新前後值？
  3. 哪些帳號或角色算是引導員？哪些算是思宜或內部人員？
  4. 引導員完成電訪的精準條件是什麼？
  5. 逾期規則是派案後 7 天、第一次聯繫後 7 天，還是其他日期基準？
  6. 信扶上傳模板的固定欄位與必填欄位有哪些？
  7. 哪些匯入欄位需要人工判斷，不能完全 mapping？
  8. 上傳後需要回寫哪些系統欄位？
  9. 是否可以先用去識別化樣本檔測試？
  10. 報告要給思宜在哪裡看最方便？
